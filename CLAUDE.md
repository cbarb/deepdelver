## What this is

**Deep Delver** — a 2.5D incremental mining game in **Godot 4.6** (GDScript). Side-view: every
mineable tile is a real 3D block viewed through an orthographic camera. 60-second runs → return to a
surface hub → spend resources/coins/skill points → dig deeper. Three co-equal mining styles:
Pickaxes (manual), Golems (auto units), Machinery (infrastructure).

Deeper context lives in three docs — read them before non-trivial work:
- `HANDOFF.md` — current implemented state, systems, simplifications, and the next planned feature (Specializations). **Most useful starting point.**
- `README.md` — milestone status and design summary.
- `Deep Delver Game Design.docx` — source-of-truth design spec (tiers, skill tree, specializations).

## Running & verifying

The game requires **Godot 4.6**. Main scene is `Main.tscn`; run with **F5** in the editor.

`godot` is **not** currently on PATH on this machine (the HANDOFF was authored on a different box) —
point it at the Godot 4.6 binary before using any command line below.

Headless verification workflow (no GUI needed):
- **Compile check:** `godot --headless --editor --quit-after 2 2>&1 | grep -iE 'SCRIPT ERROR|Parse|Compile|Failed'`
- **Behavior test:** write a throwaway `TestX.gd` + `TestX.tscn`, run `godot --headless --quit-after <frames> res://TestX.tscn`, grep the prints, then delete the temp files.
- Headless **cannot render pixels** — visual bugs (layout, colors, draw order) must be checked in the real editor; ask the user to screenshot.

**GDScript 4.6 gotcha (hit repeatedly):** walrus inference fails on comparisons —
`var x := dict.get(...) == y` mis-infers. Use an explicit type: `var x: bool = ...`.

## Architecture

Everything is built **in code** from a 1-line `Main.tscn` (a single `Node` running `Main.gd`). There are
no `.tscn` scene trees to edit — scenes are regenerated by scripts, so gameplay lives entirely in `scripts/`.

Two **autoloads** (registered in `project.godot`) hold all shared state:
- **`GameData.gd`** — all *definition* data: resources, biomes, PICKAXES/GOLEMS/MACHINES, the
  generated skill tree (`_build_skill_tree`, ~102 nodes), tile/bg/cursor asset tables. **Balancing and
  content changes go here** — add biomes/resources/upgrades/skills to these dictionaries without touching gameplay code.
- **`GameState.gd`** — persistent progression + `get_effective_stats()` (folds equipped pickaxe →
  upgrades → skills into one stat dict) + buying/leveling + save/load. `get_effective_stats()` is the
  single funnel every system reads its numbers from; new stat-affecting content plugs into `_base_stats` and this function.

Runtime flow: **`Main.gd`** is a state machine that swaps between two subtrees — `SurfaceUI` (hub) and
`MineController` (a run) — clearing and rebuilding on each transition. A run ends → `GameState.save_game()` → back to surface.

Per-file roles: `MineController.gd` (builds world/camera/lights, mouse mining, golem AI, machines,
round timer, summary), `MineGenerator.gd` (chunked grid + ore veins), `TileBlock.gd` (one textured 3D
block + hover/damage/break FX), `HUD.gd` (in-run HUD; hides OS cursor), `CursorRing.gd` (software cursor
= pickaxe sprite + cooldown ring), `SurfaceUI.gd` (shops + stockpile + start button),
`SkillTreePanel.gd` (custom-drawn radial skill tree).

## Data-model conventions that bite

- **IDs are save keys.** Save file is `user://deepdelver_save.json`; golem/skill/upgrade **ids are used
  as dictionary keys**, so renaming an id orphans existing saves. Fine in dev — `GameState.reset_progress()` wipes.
- **Golem stat keys are still prefixed `ai_*`** internally (`ai_damage`, `ai_interval`,
  `ai_resource_bonus`) — a display-only rename to "Golem" was done in the UI; don't be surprised by the mismatch.
- **Mining rule:** a tile is mineable only if ≥1 cardinal neighbour is air (surface row counts as air).
- **Overflow splash:** excess damage over a tile's HP splits evenly across the 8 surrounding blocks
  (`_overflow_splash` in `MineController.gd`). The older chain-reaction path (`_overflow_chain`/`_process_chains`)
  is **shelved but kept** behind a `SHELVED` comment — re-enable by swapping the call in `_damage_tile`.

## Assets

- `assets/tile_materials/tile_r{1..10}_c{1..7}.png` — cube-face textures (row = biome).
- `assets/biome_bg/` — backdrops `bg1..bg10` **except biome 4 = `b4.png`**.
- `assets/pickaxes2/pickaxe_{01..11}.png` — cursor sprites (01 = starter … 11 = Core).
- `tile_atlas/` — original sprite atlas, unused by the current game; kept for reference.
